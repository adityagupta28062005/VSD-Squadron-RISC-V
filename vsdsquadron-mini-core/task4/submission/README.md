# Project-6: LED Pattern Sequencer (Scriptable over UART)

## Description

This project implements an LED Pattern Sequencer scriptable over UART for the VSDSquadron Mini. Because there is no external LED available and only the standard USB cable is used, this project dynamically toggles the Onboard LED (PD6).

**The Hardware Challenge:** 
Developing this on the VSDSquadron Mini without external components presented a massive hurdle: the only available LED is hardwired to **PD6**, which is also the microcontroller's default **UART RX** pin (used for uploading code and receiving keyboard commands). 

Attempting to use standard non-blocking loops or simply driving the pin HIGH directly collided with the active USART peripheral. This caused the CH32V003 microcontroller to instantly freeze, triggered endless framing errors, and locked up the terminal whenever the pin state was mutated. 

To overcome this, we engineered a **Dynamic Pin-Multiplexing Hybrid State Machine**:

1. **Command Mode (Default)**: PD6 is initialized as `GPIO_MODE_INPUT_FLOATING`. It quietly acts as a USART listener so the user can configure patterns and delays.
2. **The Transition**: When `play` is triggered, the system strategically waits for the UART TX buffer to flush out its acknowledging messages to avoid electrical corruption. It then forcibly reinitializes PD6 as `GPIO_MODE_OUTPUT_PP`. 
3. **Playback Mode (Deaf/Output)**: The sequencer runs a blocking loop to play the bits, completely ignoring keyboard input (which would be physically impossible to read anyway while the pin is acting as a power source). This prevents internal buffer overflow crashes.
4. **Restoration & Cleanup**: After exactly 3 complete cycles, PD6 is gracefully handed back to the UART peripheral via reconfiguration. A hardware delay lets the USART settle, and a custom `while(uart_data_available())` loop flushes out any garbage electrical noise generated by the LED toggling before returning control to the user.

Getting a single pin to serve as both a high-speed interactive data listener and a visual output driver on the fly—without crashing the CH32V003—was a huge achievement for this build!

## Drivers Used

- **GPIO**: For dynamically commandeering and controlling the onboard LED (PD6).
- **UART**: For receiving configuration commands over standard PD5/PD6.
- **Timer**: A SysTick-based timekeeper generating 1ms ticks for creating accurate, non-blocking intervals.

## API Summary

### GPIO

- `gpio_init(port, pin, mode)`
- `gpio_set(port, pin)`
- `gpio_clear(port, pin)`

### UART

- `uart_init(baudrate)`
- `uart_data_available()` : Checks for newly arrived serial data without halting the CPU.
- `uart_read_char()`
- `uart_write_string(str)`

### Timer

- `timer_init()` : Starts the background 1ms interval ticker.
- `timer_get_ticks()`: Grabs the current system uptime in milliseconds.

## Build & Flash Steps

1. Navigate to `vsdsquadron-mini-core/task4/submission/`.
2. Ensure you have PlatformIO installed in VS Code.
3. Build the project: `pio run`
4. Flash the project to your board: `pio run -t upload` (or use the PlatformIO VS Code extension GUI)

## UART Settings

- Baud rate: 115200
- Data bits: 8
- Stop bit: 1
- Parity: None

## How to Demo

1. Connect your VSDSquadron mini to PC via standard USB and flash the code.
2. Open a Serial terminal (like PlatformIO serial monitor) at 115200 baud.
3. Press `Enter` to ensure the prompt `>` appears.
4. Type `pattern 111000` and press enter to register a fast-blink, slow-rest pattern.
5. Type `delay 150` and press enter.
6. Type `play` and watch the board.
7. *The terminal will lock safely, and the board will play the pattern exactly 3 times.*
8. Once finished, the terminal unlocks and shows the updated configuration banner.
9. Type `exit` to halt the sequencer when done.
